AI识卡系统检查 — AI CARD MODULE REPORT
✅ AI失败不阻塞用户 — Fallback-First设计 OK
ai-scan v4.1 采用 "Fallback-First" 设计：Gemini API失败 → 返回status="failed" + 手动录入提示；JSON解析失败 → 正则提取；甚至hash计算失败也返回fallback而非500错误。任何AI失败不影响主系统。
✅ 三级状态返回机制 OK
success (TCG, confidence≥0.5) → 自动入库; partial (NON_TCG/低confidence) → 人工确认; failed (UNKNOWN/ERROR) → 手动录入。前端总是收到200+可用数据。
✅ Gemini API降级策略 OK
主模型 gemini-2.5-flash → 429限流时降级到 gemini-2.5-flash-lite
✅ 防刷机制完整 OK
SHA-256图片hash去重 + 7天缓存 + 每日扫描次数限制 + AI成本预算 + 60秒时间戳防重放
⚠️ Edge Function verify_jwt=false — 无认证要求 HIGH
deploy_payload.json 设置 verify_jwt: false，意味着任何人无需登录即可调用ai-scan Edge Function，消耗AI成本
影响: 虽然有rate limit和budget check，但这些依赖X-User-Id头（可伪造）
⚠️ card-images 存储桶无限制 HIGH
card-images桶: public=true, file_size_limit=null, allowed_mime_types=null。card-scans桶有限制(5MB+类型检查)，但card-images桶完全开放。
影响: 任何人可上传任意大小/类型的文件到card-images桶
⚠️ 图片大小前端验证不足 MEDIUM
Edge Function仅检查 image.length < 100（太短），没有检查上限。理论上可发送数十MB的base64图片。
✅ 非卡牌图片有分类处理 OK
Gemini prompt明确分类为 TCG / NON_TCG / UNKNOWN 三类，NON_TCG返回partial需人工确认，UNKNOWN返回failed需手动录入
  • ✅ Gemini API异常处理 — 三层fallback
  • ✅ API失败不导致页面崩溃 — 前端总是收到200
  • ⚠️ 图片大小限制 — 仅检查下限100字符
  • ✅ 非卡牌图片处理 — NON_TCG分类
  • ✅ 重复识别处理 — SHA-256 hash + 7天缓存
  • ✅ 识别失败提示 — manual entry fallback
  • ❌ Edge Function无JWT认证
  • ❌ card-images桶无安全限制
