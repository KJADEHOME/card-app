认证三层防护

AdminAuth.requireAdmin() 执行流程:
  ├─ Step 1: client.auth.getSession()  ← 无会话 → 重定向 login.html
  ├─ Step 2: profiles.role 查询        ← 非 admin/super_admin → 重定向 index.html
  └─ Step 3: rpc('require_admin')      ← RPC 抛错 → 重定向 index.html (fail-closed)

所有3步均在前端 init() 函数中执行，页面内容在认证通过前不会渲染。
RPC require_admin() 为 SECURITY DEFINER，使用 auth.uid() 服务端鉴权，前端无法伪造。

Fail-closed 机制

  • ✅ Supabase SDK 未加载 → document.body.replaceChildren() + throw
  • ✅ RiskControl 未加载 → document.body.replaceChildren() + throw
  • ✅ require_admin() RPC 异常 → 重定向 index.html（不放行）

结论: 全部 7 个管理员页面统一使用 AdminAuth.requireAdmin()，三层认证（Auth会话 + profiles.role + require_admin RPC），fail-closed 设计完整。

async function logout() {
    // 1. 注销 Supabase Auth 会话 (服务端失效)
    await client.auth.signOut();

审计项 6: 非管理员/商户访问拦截

PASS

普通用户和 merchant 无法进入后台

客户端拦截 (快速路径)

AdminAuth.check() 流程:
  ├─ auth.getSession() → 无会话 → { isAdmin: false }
  ├─ profiles.role 查询 → role !== 'admin' && !== 'super_admin' → { isAdmin: false }
  └─ isAdmin === false → requireAdmin() 重定向到 index.html

profiles.role 可选值: user | merchant | admin | super_admin

仅 admin 和 super_admin 通过客户端检查。user 和 merchant 被拦截。

服务端拦截 (权威路径)

require_admin() RPC (SECURITY DEFINER):
  v_uid := auth.uid();
  IF v_uid IS NULL → RAISE EXCEPTION 'FORBIDDEN'
  SELECT role INTO v_role FROM profiles WHERE id = v_uid;
  IF v_role NOT IN ('admin', 'super_admin') → RAISE EXCEPTION 'FORBIDDEN'
  RETURN v_uid;

即使前端被篡改绕过客户端检查，RPC 仍会在服务端拒绝非管理员调用。

登录流程验证

步骤操作非管理员结果
1db.auth.signInWithPassword()Supabase Auth 登录成功（仅认证，不鉴权）
2AdminAuth.requireAdmin({ redirect: false })返回 null（role 不是 admin）
3db.auth.signOut()立即注销会话
4显示错误消息"该账号没有管理员权限"

RLS 防护

  • ✅ profiles 表: auth.uid() = id — 用户只能读自己的 profile
  • ✅ admins 表: 仅 SELECT，无写权限
  • ✅ admin_audit_logs: 仅管理员可读，前端禁止写
  • ✅ 管理员 RPC 内部调用 require_admin()，非管理员调用抛出 FORBIDDEN
  • ✅ set_user_role(): 仅 super_admin 可调用，普通 admin 也被拒绝

结论: 普通用户和 merchant 在客户端和服务端两个层面均被拦截。三层防护（客户端 profiles.role 检查 + 服务端 require_admin() RPC + RLS 策略）确保非管理员无法访问任何后台功能。
